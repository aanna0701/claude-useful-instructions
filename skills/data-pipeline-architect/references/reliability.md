# 파이프라인 신뢰성 원칙 — 상세 정의

이 문서는 Phase 1에서 구조 원칙(8원칙) 진단과 함께 참조하는 **실행 신뢰성** 진단 기준이다.
데이터 구조가 올바르더라도 파이프라인 실행의 신뢰성이 없으면 운영에서 무결성이 깨진다.

각 원칙마다 **검증 질문, 위반 신호, 구현 패턴, 체크리스트**를 포함한다.

---

## 원칙 R1: 멱등성 (Idempotency)

동일한 데이터를 여러 번 처리해도 결과가 항상 같아야 한다.
장애 복구 시 특정 구간을 재실행(Retry)할 때 데이터 중복이나 왜곡을 방지한다.

**검증 질문:** 같은 입력으로 같은 단계를 2번 실행하면 결과가 동일한가?

**위반 신호:**
- INSERT만 사용하고 중복 방지 로직이 없음
- 카운터나 시퀀스가 재실행 시 증가함
- 파일 append 방식으로 재실행 시 데이터가 2배로 늘어남
- "한 번만 실행해야 합니다" 같은 주석이나 문서가 있음

**구현 패턴:**
- `UPSERT` (Insert or Update) 구문으로 중복 삽입 방지
- 처리 전 대상 파티션을 삭제하고 다시 쓰기 (delete-then-write)
- progress tracker로 완료된 항목 스킵
- 자연키(natural key) 또는 content hash 기반 중복 탐지

**체크리스트:**
- [ ] 모든 단계가 재실행 시 동일 결과를 보장하는가?
- [ ] INSERT 대신 UPSERT를 사용하거나 동등한 중복 방지가 있는가?
- [ ] 파일 출력 시 append가 아닌 overwrite/atomic-rename을 사용하는가?

## 원칙 R2: 데이터 계약과 스키마 관리 (Data Contract)

소스와 타겟 간 데이터 형식(Schema)에 대한 명시적 계약이 있어야 한다.
비정상 데이터는 파이프라인 전체를 오염시키지 않고 격리되어야 한다.

**검증 질문:** 예상과 다른 데이터가 유입되면 어떻게 되는가?

**위반 신호:**
- 스키마 검증 없이 데이터를 그대로 통과시킴
- 스키마 변경 시 하류 단계가 무경고로 깨짐
- 비정상 레코드가 정상 데이터와 섞여 저장됨
- "이 필드가 언제부터 추가됐지?" 추적이 불가능

**구현 패턴:**
- **Schema Registry**: Pydantic BaseModel, JSON Schema, 또는 Avro/Protobuf로 계약 정의
- **Dead Letter Queue (DLQ)**: 스키마 위반 레코드를 별도 큐/디렉토리로 격리
- **버전 관리**: 스키마에 버전 번호를 부여하고 하위 호환성 규칙 적용
- **진입점 차단**: 계약 위반 데이터는 즉시 차단, DLQ로 이동

**체크리스트:**
- [ ] 모든 단계 간 입출력에 명시적 스키마 계약이 있는가?
- [ ] 스키마 위반 데이터가 DLQ로 격리되는가? (전체 파이프라인 중단 없이)
- [ ] 스키마 버전 관리가 있어 하위 호환성을 검증할 수 있는가?

---

## 원칙 R3: 원자성과 트랜잭션 관리 (Atomicity)

하나의 논리적 단위 내에서 모든 작업이 성공하거나, 하나라도 실패하면 전체가 취소(Rollback)되어야 한다.
부분 완료 상태가 남아 데이터를 오염시키는 것을 방지한다.

**검증 질문:** 단계 실행 중간에 장애가 발생하면 부분 결과가 남는가?

**위반 신호:**
- 최종 저장소에 직접 쓰기(direct write)하여 중간 실패 시 불완전 데이터 노출
- 트랜잭션 없이 여러 테이블을 순차 업데이트
- 장애 복구 후 "어디까지 처리됐는지" 확인할 방법이 없음
- 스트리밍 처리에서 오프셋/체크포인트가 없음

**구현 패턴:**
- **Staging Area**: 임시 공간에서 처리 완료 후 atomic rename/swap으로 최종 반영
- **DB 트랜잭션**: 여러 테이블 업데이트를 단일 트랜잭션으로 묶기
- **Checkpointing**: 스트리밍/배치 처리 시 진행 상태를 주기적으로 기록
- **Write-Audit-Publish**: staging에 쓰기 → 검증 → 통과 시 publish

**체크리스트:**
- [ ] 각 단계가 중간 실패 시 불완전 결과를 남기지 않는가?
- [ ] Staging area 또는 트랜잭션으로 원자적 커밋이 보장되는가?
- [ ] 장애 복구 시 마지막 성공 지점부터 재시작할 수 있는가? (checkpoint)

---

## 원칙 R4: 모니터링과 가시성 (Observability)

데이터가 흐르는 과정에서 무결성이 깨졌는지 실시간으로 파악할 수 있어야 한다.
문제 발생 시 소스까지 역추적이 가능해야 한다.

**검증 질문:** 특정 데이터의 오류를 발견했을 때 소스까지 역추적이 가능한가?

**위반 신호:**
- 파이프라인 실행 로그가 없거나 불충분
- 입력 건수와 출력 건수를 비교할 방법이 없음
- 데이터 지연이나 처리 속도 저하를 감지할 수 없음
- "이 잘못된 데이터가 언제 어디서 유입됐는지" 추적 불가

**구현 패턴:**
- **Data Lineage**: 데이터의 생성-변환-소멸 경로 추적 (Lineage Tracker 에이전트)
- **Audit Logging**: 각 단계의 입력/출력 건수, 처리 시간, 에러 수 기록
- **Count Reconciliation**: 단계 간 입력 건수 = 출력 건수 + 스킵 + 에러 검증
- **Health Check**: 처리 지연(Latency), 처리량(Throughput) 모니터링 + 임계값 경고

**체크리스트:**
- [ ] 각 단계에서 입출력 건수가 기록되고 reconciliation이 가능한가?
- [ ] 최종 출력의 각 레코드에서 원본까지 역추적이 가능한가?
- [ ] 처리 지연이나 에러율 급증 시 경고가 발생하는가?

---

## 원칙 R5: 품질 검증 (Data Quality Validation)

단순 전송을 넘어 데이터 내용 자체의 무결성을 검사한다.
잘못된 데이터가 하류로 전파되는 것을 조기에 차단한다.

**검증 질문:** 형식은 맞지만 내용이 비정상인 데이터를 걸러낼 수 있는가?

**위반 신호:**
- 스키마만 통과하면 무조건 다음 단계로 전달됨
- 필수 필드의 NULL 비율을 모니터링하지 않음
- 값의 범위나 분포에 대한 검증이 없음
- 테이블 간 참조 무결성을 런타임에 확인하지 않음

**구현 패턴:**
- **Null Check**: 필수 필드의 NULL/빈값 비율이 임계값 이하인지 검증
- **Range & Format Check**: 숫자 범위, 날짜 형식, 문자열 패턴 검증 (예: 나이 0~150)
- **Consistency Check**: 외래키 관계, 비즈니스 룰 일관성 검증 (예: 시작일 ≤ 종료일)
- **Distribution Check**: 값 분포의 급격한 변화 탐지 (이전 배치 대비 통계적 비교)
- **Freshness Check**: 데이터 타임스탬프가 허용 범위 내인지 검증

**체크리스트:**
- [ ] 필수 필드의 NULL 비율에 대한 임계값이 있는가?
- [ ] 숫자/날짜 필드에 범위 검증이 있는가?
- [ ] 테이블 간 참조 무결성을 런타임에 확인하는가?
- [ ] 이전 배치 대비 급격한 분포 변화를 탐지하는가?

## 진단 출력 형식

Phase 1 구조 진단(8원칙)과 함께 아래 형식으로 출력한다:

```markdown
## 파이프라인 신뢰성 진단

| # | 원칙 | 판정 | 위험 구간 | 처방 |
|---|------|------|-----------|------|
| R1 | 멱등성 | ⚠️ | Stage 2 INSERT only | UPSERT로 변경, progress tracker 추가 |
| R2 | 데이터 계약 | ⚠️ | Stage 1→2 경계 | DLQ 추가, 스키마 버전 관리 도입 |
| R3 | 원자성 | ✅ | - | - |
| R4 | 가시성 | ⚠️ | 전체 | audit log + count reconciliation 추가 |
| R5 | 품질 검증 | ⚠️ | Stage 3 출력 | null check + range check 추가 |

### 종합 소견
[2~3문장으로 신뢰성 강점과 핵심 개선 포인트 요약]
```

## 설계 체크리스트 (Phase 5에 포함)

```
□ 재시도 가능성: 실패 시 동일 시점에서 재시작해도 데이터가 중복되지 않는가?
□ 격리성: 비정상 데이터가 유입되었을 때 전체 파이프라인이 중단되지 않고 격리되는가?
□ 추적 가능성: 특정 데이터의 오류를 발견했을 때 소스까지 역추적이 가능한가?
□ 확장성: 데이터 양이 늘어나도 무결성 검증 로직이 병목이 되지 않는가?
```
