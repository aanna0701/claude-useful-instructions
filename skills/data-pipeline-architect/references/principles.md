# 8 불변 원칙 — 상세 정의

이 문서는 Phase 1 진단 시 참조하는 원칙 상세 정의다.
각 원칙마다 **검증 질문, 위반 신호, 위반 예시, 처방 패턴**을 포함한다.

---

## 원칙 1: Single Source of Truth (단일 정본)

같은 정보는 시스템 전체에서 정확히 한 곳에만 존재해야 한다.

**검증 질문:** 같은 정보가 2곳 이상에 저장되는가?

**위반 신호:**
- DB 컬럼과 파일명에 같은 메타정보가 인코딩됨
- 여러 테이블에 동일 필드가 중복 존재
- 설정값이 코드와 config 파일 양쪽에 하드코딩됨

**위반 예시:**
- `user.email`과 `order.customer_email`이 둘 다 존재
- 파일명이 `20260310_vasco_v1_angiography.csv`인데 DB에도 같은 정보가 컬럼으로 있음
- 이미지 크기가 코드의 상수와 config YAML 양쪽에 정의됨

**처방 패턴:**
- 하나를 정본(source of truth)으로 지정하고, 나머지는 참조(FK) 또는 제거
- 파일명에는 식별자만, 메타정보는 DB에만 저장
- 설정값은 config 한 곳에서만 관리, 코드는 config를 읽기만

---

## 원칙 2: 의미의 단일성 (No Gray Zone)

하나의 필드, 테이블, 파일은 정확히 하나의 의미만 가져야 한다.

**검증 질문:** 하나의 요소가 상황에 따라 다른 의미를 갖는가?

**위반 신호:**
- 같은 컬럼이 행에 따라 다른 단위나 의미를 가짐
- 하나의 테이블이 성격이 다른 엔티티를 혼합 저장
- 플래그 필드가 3개 이상의 의미를 오버로딩

**위반 예시:**
- `amount` 필드가 어떤 행에서는 매출, 어떤 행에서는 환불, 어떤 행에서는 수수료
- `events` 테이블에 클릭/구매/환불이 모두 같은 스키마로 섞여 있음
- `status` 필드가 0=대기, 1=진행, 2=완료, 3=에러, 4=취소를 한 정수로 표현

**처방 패턴:**
- 의미별로 컬럼 또는 테이블 분리
- type/category 컬럼으로 명시적 구분 + CHECK 제약
- enum 타입 사용 또는 별도 참조 테이블

---

## 원칙 3: 자기 설명적 명명 (Self-Describing)

이름만 보고 의미와 단위를 즉시 알 수 있어야 한다.

**검증 질문:** 3개월 후 이 이름만 보고 의미를 알 수 있는가?

**위반 신호:**
- 약어만으로 구성된 이름 (col_a, tbl_01, val1)
- 단위가 이름에 없는 숫자 필드 (speed, distance, angle)
- 의미가 모호한 범용 이름 (data, info, tmp, result)

**위반 예시:**
- `f3_rot` → 무엇의 rotation인지, 단위가 뭔지 알 수 없음
- `velocity` → 선속도? 각속도? 단위는 m/s? mm/s?
- `process()` 함수 → 무엇을 어떻게 process하는지 알 수 없음

**처방 패턴:**
- `{도메인}_{측정대상}_{단위}` 패턴: `wire_rot_vel_rad_s`
- 테이블명은 단수 명사 + 엔티티 이름: `episode`, `frame`, `daily_metric`
- 약어 사용 시 프로젝트 용어집(glossary)에 정의 필수

---

## 원칙 4: 명확한 경계 (Boundary)

각 테이블, 모듈, 파일의 책임 범위가 명확하고 겹치지 않아야 한다.

**검증 질문:** 이 데이터가 A 소관인지 B 소관인지 애매한 곳이 있는가?

**위반 신호:**
- 하나의 테이블에 서로 다른 생명주기를 가진 데이터가 혼재
- 변경 사유가 2개 이상인 테이블 (A 기능 변경 시에도, B 기능 변경 시에도 수정 필요)
- "이 컬럼 여기 넣을까 저기 넣을까" 고민이 반복됨

**위반 예시:**
- `order` 테이블에 배송상태, 결제상태, 리뷰점수가 모두 존재
- `user` 테이블에 인증정보, 프로필, 결제수단, 설정이 모두 한 곳에
- 센서 데이터 테이블에 센서 메타정보와 측정값이 혼재

**처방 패턴:**
- 하나의 테이블은 하나의 엔티티만 담도록 분리
- 생명주기가 다르면 반드시 분리 (자주 바뀌는 것 vs 거의 안 바뀌는 것)
- "이 테이블을 변경해야 하는 이유"가 2가지 이상이면 분리 신호

---

## 원칙 5: 변화 대응 설계 (Design for Change)

현재값만이 아니라, 값이 바뀔 가능성에 대비해야 한다.

**검증 질문:** 시간에 따라 바뀔 수 있는 값이 현재값만 저장되어 있는가?

**위반 신호:**
- 등급, 상태, 가격 등 변경 가능한 값이 단일 필드로만 저장
- "이전 값이 뭐였지?" 질문에 대답할 수 없는 구조
- 스키마 변경 시 마이그레이션 계획이 없음

**위반 예시:**
- 고객 등급을 `user.tier = 'gold'`로만 저장 (변경 이력 없음)
- 상품 가격이 `product.price`만 있어서 가격 변경 전 주문 확인 불가
- 로봇 config가 바뀌었는데 이전 config로 수집한 데이터 재현 불가

**처방 패턴:**
- 변경 가능한 값은 이력 테이블 또는 SCD(Slowly Changing Dimension) 적용
- `valid_from`, `valid_to` 패턴으로 시점 조회 가능하게
- config는 버전 관리하고, 데이터에 사용된 config 버전을 기록

---

## 원칙 6: NULL 의미 정의

모든 nullable 필드에서 NULL의 의미가 명확히 정의되어야 한다.

**검증 질문:** 이 필드가 NULL일 때, "아직 모름" / "해당 없음" / "입력 실수" 중 어느 것인가?

**위반 신호:**
- nullable 필드에 대한 문서나 주석이 없음
- 코드에서 `IS NULL` 체크 후 다른 로직으로 분기하는 곳이 많음
- 분석 시 NULL을 어떻게 처리해야 할지 사람마다 다르게 해석

**위반 예시:**
- `notes = NULL` → 메모를 안 쓴 건지, 메모 기능이 없던 시절 데이터인지 불명
- `score = NULL` → 0점인지, 아직 채점 안 된 건지, 채점 대상이 아닌 건지
- `parent_id = NULL` → 루트 노드인지, 관계 설정이 누락된 건지

**처방 패턴:**
- 스키마 주석 또는 문서에 각 nullable 필드의 NULL 의미 명시
- "해당 없음"과 "아직 모름"이 둘 다 필요하면 별도 상태 필드 추가
- 가능하면 NOT NULL + DEFAULT로 NULL 자체를 제거

---

## 원칙 7: 적정 단순성 (No Simpler Than Necessary)

가능한 한 단순하게, 그러나 필요 이상으로 단순하게는 안 된다.

**검증 질문:** 예측 가능한 분리 니즈가 있는데 과도하게 합쳐놓은 곳은? 반대로 불필요하게 쪼개진 곳은?

**위반 신호 (과소 구조화):**
- 주소가 단일 TEXT 필드 → 시/구/동별 분석 불가
- JSON blob에 고정 스키마 데이터를 넣음 → 쿼리 비효율
- 여러 값을 콤마로 이어붙인 문자열

**위반 신호 (과잉 구조화):**
- 사용 빈도가 극히 낮은 데이터를 위해 별도 테이블 생성
- 1:1 관계인데 불필요하게 테이블을 분리
- 5개도 안 되는 enum 값을 위해 참조 테이블 생성

**처방 패턴:**
- "이 필드를 나중에 분리해서 쿼리할 확률"로 판단
- 확률 높으면 미리 구조화, 낮으면 단일 필드 유지
- JSON은 스키마가 진짜 유동적일 때만 사용

---

## 원칙 8: 관계 명시 (Explicit Cardinality)

엔티티 간 관계의 종류와 방향이 스키마 수준에서 강제되어야 한다.

**검증 질문:** A와 B의 관계가 1:1인지 1:N인지 N:M인지 스키마만 보고 알 수 있는가?

**위반 신호:**
- FK가 없고 코드에서만 JOIN 조건으로 관계를 표현
- 1:N인데 JSON 배열로 N쪽을 저장 (정규화 위반)
- N:M인데 junction table 없이 콤마 구분 문자열

**위반 예시:**
- user와 project 관계가 코드의 WHERE절에서만 표현됨
- `order.product_ids = "101,102,103"` (콤마 구분 문자열)
- `config.sensors = [{"id": 1}, {"id": 2}]` (JSON으로 1:N 표현)

**처방 패턴:**
- FK 제약 조건으로 관계를 스키마 수준에서 강제
- N:M은 반드시 junction table
- 1:N은 자식 테이블에 FK, 부모는 자식을 직접 참조하지 않음
