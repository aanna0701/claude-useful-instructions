# How-to Guide 작성 규칙

`doc-writer-howto` 에이전트가 문서 작성 전 반드시 Read하는 규칙.

---

## 정체성

How-to Guide = **문제 해결(Problem-solving)**을 위한 문서.
독자는 이미 기본을 알고, **특정 문제를 해결**하려 한다.
Tutorial처럼 손잡아주지 않고, Reference처럼 전부 나열하지 않는다.

---

## DO / DON'T

| DO | DON'T |
|----|-------|
| **문제 중심** 제목 ("~하는 방법") | 기능 중심 제목 ("X 기능 소개") |
| **유연성** 제공 (독자 환경 다를 수 있음) | Tutorial처럼 고정 경로만 제시 |
| **전제 조건** 명시 | 기초부터 가르치기 (→ Tutorial) |
| 절차를 **간결하게** 유지 | "왜"를 장황하게 설명 (→ Explanation) |
| 실전에서 바로 쓸 수 있게 | 교육용 예제로 대체 |

---

## 문서 구조 템플릿

```markdown
# [~하는 방법] / [How to ~]

> [이 가이드가 해결하는 문제 한 문장]

## 전제 조건
- [사전 지식 / 완료해야 할 Tutorial 링크]
- [필요한 도구 + 최소 버전]
- [필요한 권한 / 환경]

## 절차

### 1. [동사로 시작]
\```bash
명령어 또는 코드
\```
> 💡 [환경에 따라 달라질 수 있는 부분]

### 2. [동사로 시작]
...

## 검증
[결과가 올바른지 확인하는 방법]

## 참고
- [관련 Reference 링크]
- [관련 Explanation 링크]
```

---

## 핵심 규칙

### 1. 제목은 "~하는 방법" 패턴
- "인증 토큰을 갱신하는 방법"
- "프로덕션에서 마이그레이션을 롤백하는 방법"
- 제목만 보고 "이게 내 문제의 답인가?" 판단 가능해야 함.

### 2. 전제 조건 게이트
- 자격 미달 독자는 Tutorial로 보낸다.
- "이 가이드는 [Tutorial 제목]을 완료한 독자를 대상으로 합니다."

### 3. 유연성 허용 (Tutorial과의 핵심 차이)
- Tutorial은 선택지를 제거. How-to Guide는 **선택지를 인정**.
- "PostgreSQL이면 `pg_dump`, MySQL이면 `mysqldump`를 쓰세요."
- 선택지 3개 초과 시 표로 정리.

### 4. 설명은 최소한
- "왜"는 1문장 이내. 깊은 배경 → Explanation 링크.
- How-to에서 가장 흔한 실수: 절차 중간에 아키텍처 설명 삽입.

### 5. 실전 지향
- 프로덕션에서 바로 쓸 수 있는 수준의 예제 코드.
- `foo`, `bar` 대신 현실적인 값. 에러 핸들링 포함.

### 6. 분량과 범위
- 하나의 How-to = 하나의 문제.
- "인증 설정 및 모니터링 구성" → 2개로 분리.
- 5~8단계 이상적. 12단계 초과 시 분할 고려.

---

## 톤 & 스타일

- 직접적이고 실무적: "~하세요", "~를 실행합니다"
- 독자 역량을 신뢰. 기초 재설명 안 함.
- 결과 지향: "~를 달성하려면 이렇게 합니다"

---

## Anti-Patterns

1. **"교과서형"**: 절차보다 이론이 더 많음
2. **"만능 가이드"**: 하나의 문서에서 3개 이상 문제 해결
3. **"Tutorial 위장"**: "먼저 Python을 설치합니다"부터 시작
4. **"Reference 위장"**: 모든 옵션의 전체 목록 나열
