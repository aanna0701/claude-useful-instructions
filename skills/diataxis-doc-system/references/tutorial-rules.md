# Tutorial 작성 규칙

`doc-writer-tutorial` 에이전트가 문서 작성 전 반드시 Read하는 규칙.

---

## 정체성

Tutorial = **학습(Learning)**을 위한 문서.
독자는 이 주제를 **처음** 접하는 사람.
문서를 끝까지 따라하면, 독자는 **동작하는 무언가**를 손에 쥐게 된다.

---

## DO / DON'T

| DO | DON'T |
|----|-------|
| 독자를 **손잡고 이끌어라** | 선택지를 주지 마라 ("A를 쓰거나 B를 쓸 수도") |
| **모든 단계**를 빠짐없이 보여라 | "자명한" 단계를 생략하지 마라 |
| **즉시 확인 가능한 결과**를 매 단계마다 제공 | 결과 없이 10단계 연속 나열 금지 |
| **최소한의 설명**만 곁들여라 | 배경 이론 장황하게 풀지 마라 (→ Explanation) |
| 구체적인 **예제 값** 제공 | `<your-value-here>` 플레이스홀더 남기지 마라 |

---

## 문서 구조 템플릿

```markdown
# [Tutorial 제목]: [독자가 만들게 될 것]

> 이 튜토리얼을 마치면 [구체적 결과물]을 갖게 됩니다.
> 예상 소요 시간: [N]분

## 사전 준비
- [도구 1] — 설치 확인: `명령어`
- [도구 2] — 설치 확인: `명령어`

## 1단계: [동사로 시작]
[왜 이 단계가 필요한지 1문장]
\```bash
구체적_명령어 --with=실제값
\```
✅ **확인**: [성공 시 보여야 하는 것]

## 2단계: [동사로 시작]
...

## N단계: 결과 확인
[최종 결과물 출력 예시]

## 다음 단계
- [How-to Guide 링크]: 이제 [심화 작업]을 해보세요.
- [Explanation 링크]: [방금 한 것]이 왜 그렇게 동작하는지 알고 싶다면.
```

---

## 핵심 규칙

### 1. 황금 경로(Golden Path)
- 선택지를 제거. 독자에게 결정을 요구하지 마라.
- OS별 분기 필요 시 **탭/토글**로 감추되, 기본값 하나만 펼침.
- "당신의 상황에 따라 다릅니다"는 Tutorial에서 금지어.

### 2. 체크포인트 패턴
- **매 단계 끝**에 `✅ 확인:` 블록.
- 예: "터미널에 `Server running on port 3000`이 보이면 성공입니다."

### 3. 코드 블록
- 복사-붙여넣기 가능해야 함. 플레이스홀더 최소화.
- 불가피한 플레이스홀더: `YOUR_API_KEY` (대문자_언더스코어).
- 터미널 출력과 입력 구분 (`$` 프롬프트 또는 별도 블록).

### 4. 실패 경로
- 자주 발생하는 오류 1~2개만 `⚠️ 문제 발생 시:` 박스.
- 드문 오류는 Troubleshooting How-to Guide로 분리.

### 5. 분량
- 총 10단계 이내 목표. 초과 시 Part 1/2로 분할.
- 각 단계는 하나의 동작(action)만 포함.

---

## 톤 & 스타일

- 2인칭: "먼저 ~~를 설치합니다" (O), "사용자는 ~~를 설치해야 합니다" (X)
- 현재 시제: "파일이 생성됩니다" (O), "파일이 생성될 것입니다" (X)
- 격려 톤: "잘 되고 있습니다. 이제 마지막 단계입니다."

---

## Anti-Patterns

1. **"설명 튜토리얼"**: 단계 중간에 3문단짜리 이론 삽입
2. **"레퍼런스 튜토리얼"**: 모든 옵션과 파라미터 나열
3. **"선택형 튜토리얼"**: "A 또는 B를 선택하세요" 반복
4. **"블라인드 튜토리얼"**: 10단계 중 중간 확인 0회
